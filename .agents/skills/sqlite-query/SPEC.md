# Spec: sqlite-query

## Objective
Позволяет агенту выполнять SQL-запросы к SQLite-базам данных по текстовому описанию пользователя. Пользователь говорит «сколько статей в кеше?», агент разбирается со схемой, пишет SQL и показывает результат. Без навыка агент либо отказывается, либо пытается угадать данные.

## Skill Type
- **Type:** technique
- **Category:** productivity

## Invocation
- **Invocation:** model-invoked
- **Triggers:** пользователь упоминает SQLite-файл (.db), базу данных, кеш, «сколько записей», «покажи данные», «выполни запрос» в контексте файловой БД

## Overview
Агент получает от пользователя путь к SQLite-файлу и текстовый запрос. Он самостоятельно изучает схему (таблицы, колонки, типы, связи), формулирует SQL, выполняет через `sqlite3` CLI и возвращает результат в табличном виде с кратким текстовым резюме. Никакого дополнительного кода или скриптов не требуется — только инструкции в SKILL.md.

## When to Use
- Пользователь явно указал путь к `.db` / `.sqlite` / `.sqlite3` файлу
- Пользователь спросил о данных, которые заведомо лежат в SQLite (например, «сколько статей в кеше hackaday»)
- Пользователь сказал «выполни SQL-запрос» или «напиши запрос к базе»
- Пользователь предоставил текстовое описание того, что хочет найти

**Когда НЕ использовать:**
- База данных — не SQLite (PostgreSQL, MySQL и т.д.) — это другой скилл
- Пользователь явно дал готовый SQL-запрос (просто выполнить — не нужен NL→SQL перевод)
- Данные уже доступны в другом формате (CSV, JSON) — скилл для SQLite

## Core Process / Workflow

### Шаг 1: Получить путь к БД
Если пользователь не указал путь явно — найти `.db`/`.sqlite`-файлы в рабочей директории или директории, относящейся к задаче. Если найден один — взять его. Если несколько — спросить пользователя, какой использовать.

**Completion Criterion:** Путь к БД известен и файл существует.

### Шаг 2: Исследовать схему
Выполнить последовательно:
1. `.tables` — список всех таблиц
2. `.schema <table_name>` — DDL для каждой таблицы (начинать с той, что вероятно относится к запросу)
3. `PRAGMA table_info(<table_name>);` — колонки, типы, NULL, default
4. `SELECT * FROM <table_name> LIMIT 3;` — пример данных, чтобы понять наполнение

Команды выполнять через `sqlite3 <path> ".tables"`, `sqlite3 <path> ".schema"` и т.д.

**Completion Criterion:** Агент знает названия всех таблиц, их колонки и типы данных, а также имеет образец данных.

### Шаг 3: Сформулировать SQL
На основе схемы и NL-запроса пользователя написать SQL. Проверить:
- Имена таблиц и колонок соответствуют реальным (из шага 2)
- Типы данных учтены (даты как даты, числа как числа)
- JOIN используется только когда нужны данные из нескольких таблиц
- Агрегации (COUNT, SUM, AVG, GROUP BY) соответствуют вопросу

Если запрос неоднозначен — предложить 2-3 варианта и спросить пользователя.

**Completion Criterion:** SQL-запрос написан и отражает намерение пользователя.

### Шаг 4: Выполнить и показать результат
```bash
sqlite3 -header -column "<path>" "<SQL>"
```

Если результат большой (>20 строк) — показать `LIMIT 20` и сообщить общее количество строк через `COUNT(*)`.

**Completion Criterion:** Результат выведен пользователю.

### Шаг 5: Интерпретировать
Дать краткое (1-3 предложения) текстовое резюме результата: что нашли, сколько строк, ключевые значения.

**Completion Criterion:** Пользователь получил ответ на свой вопрос.

### Шаг 6: Уточнить
Спросить «Всё ли верно? Нужно уточнить запрос?» и при необходимости повторить шаги 3-6.

**Completion Criterion:** Пользователь подтвердил, что ответ верен, или отказался от уточнений.

## Completion Criteria

- [ ] Пользователь получил таблицу с данными и текстовое резюме

## Common Rationalizations

| Rationalization | Reality |
|-----------------|---------|
| «Я знаю эту БД, схему смотреть не нужно» | Имена колонок могли измениться, добавились новые таблицы. Схема — единственный source of truth. |
| «Запрос простой (SELECT *), можно без схемы» | Даже `SELECT *` требует знать имя таблицы. |
| «Достаточно одной таблицы, остальные не важны» | Пользователь мог иметь в виду связанные данные. Проверить все таблицы. |
| «Выполню без проверки, если ошибка — исправлю» | Каждая ошибка — потеря времени и контекста. Лучше проверить SQL мысленно перед запуском. |
| «Покажу сырой SQL, пользователь сам разберётся» | Задача скилла — перевести NL→SQL→результат. Показывать SQL можно, но не вместо результата. |

## Red Flags

- Агент пишет SQL, **не прочитав** `.schema` и `PRAGMA table_info`
- Агент использует `SELECT *` когда явно нужны конкретные колонки
- Агент не показывает результат (только SQL)
- Агент выполняет `INSERT`/`UPDATE`/`DELETE`/`DROP` без явного разрешения пользователя
- Агент не обрабатывает ошибку SQL (например, «no such table») — не исправляет запрос
- Агент не показывает LIMIT при большом результате

## Verification

- [ ] По `.db`-файлу из hackaday удалось ответить «сколько статей» (подтверждено вручную)
- [ ] Агент исследовал схему перед SQL — проверить лог вызовов
- [ ] Агент корректно выполнил COUNT, WHERE с датой, GROUP BY, JOIN
- [ ] Агент спросил разрешение перед модифицирующим запросом

## Boundaries
- **Always do:** исследовать схему; использовать `-header -column`; показывать LIMIT для больших результатов
- **Ask first:** INSERT, UPDATE, DELETE, DROP, ALTER — любые модифицирующие запросы
- **Never do:** выполнять модифицирующие запросы без подтверждения; угадывать имена таблиц; показывать сырой SQL без результата

## Test Strategy

### Skill Type → Test Method
**Technique:** Application scenarios.

### Test Scenarios (execute against hackaday.db)
1. **Простой COUNT:** «Сколько всего статей?» → `SELECT COUNT(*) FROM articles`
2. **WHERE с категорией:** «Сколько статей с категорией LED Hacks?» → `SELECT COUNT(*) FROM articles WHERE category = 'LED Hacks'`
3. **Агрегация + GROUP BY:** «Сколько статей по каждой категории?» → `SELECT category, COUNT(*) FROM articles GROUP BY category`
4. **Фильтр по дате:** «Статьи за март 2026» → `SELECT * FROM articles WHERE date LIKE '2026-03%'`
5. **JOIN:** «Покажи статьи с analysis_scores» → `SELECT * FROM articles JOIN analysis_scores ON articles.id = analysis_scores.article_id LIMIT 10`
6. **Неоднозначность:** «Достань данные о комментариях» → должен посмотреть таблицу `comments` и предложить варианты
7. **Ошибка:** Запросить несуществующую таблицу → агент должен исправить

### Key Rationalizations to Bulletproof
- «Я знаю схему» — в начале каждого выполнения загружать схему заново
- «Запрос простой» — даже простой запрос требует валидации имён

### Baseline Failure (without skill)
Без скилла агент:
- Отвечает «я не могу работать с SQLite» или предлагает написать Python-скрипт
- Пытается угадать данные по памяти (нет, тестировалось: модель выдаёт вымышленные цифры)
- Пишет SQL вслепую и получает ошибку «no such table»
