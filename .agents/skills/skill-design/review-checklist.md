# Skill Specification Review Checklist

Используется в Фазе 3 (Quality) для проверки готовой спецификации скилла.

## 1. Placeholder Scan
- [ ] Нет «TBD», «TODO», «implement later», «fill in details»
- [ ] Нет «Add appropriate error handling» / «add validation» / «handle edge cases» без конкретики
- [ ] Нет «Write tests for the above» без указания, какие именно

## 2. Internal Consistency
- [ ] Секции не противоречат друг другу
- [ ] Architecture соответствует описанию фич
- [ ] Типы и сигнатуры согласованы
- [ ] Completion criteria достижимы при описанном процессе

## 3. Scope Check
- [ ] Спецификация фокусируется на одном чётком скоупе
- [ ] Не покрывает несколько независимых подсистем
- [ ] Если scope слишком велик — предложить декомпозицию

## 4. Ambiguity Check
- [ ] Каждое требование можно интерпретировать только одним способом
- [ ] Все assumption явно перечислены
- [ ] Нет оценок и суждений без конкретики («качественный», «быстрый», «надёжный»)

## 5. YAGNI
- [ ] Нет незапрошенных фич
- [ ] Нет over-engineering
- [ ] Spec решает реальную проблему, а не гипотетическую

## 6. No-Op Test
- [ ] Каждая инструкция меняет поведение относительно дефолта модели
- [ ] Нет строк, которые модель уже делает по умолчанию
- [ ] Если leading word слишком слабый — заменить на более сильный

## 7. Leading Word Audit
- [ ] Есть компактное понятие, которое якорит поведение
- [ ] Leading word рекрутирует priors модели (не надо определять с нуля)
- [ ] Leading word используется консистентно по всему скиллу

## 8. Duplication Check
- [ ] Каждое значение ровно в одном месте (Single Source of Truth)
- [ ] Нет повторяющихся паттернов между разными секциями
- [ ] Триггеры в description не дублируют друг друга

## 9. Progressive Disclosure
- [ ] SKILL.md не превышает ~500 строк
- [ ] Reference материал вынесен в supporting files
- [ ] Ссылки только одного уровня глубины от SKILL.md

## 10. Rationalization Completeness
- [ ] Каждый skip-worthy step имеет counter-argument
- [ ] Рационализации основаны на реальных тестах, а не гипотезах
- [ ] Red Flags покрывают observable признаки нарушения

## 11. User Review Gate
- [ ] Спецификация показана пользователю
- [ ] Пользователь утвердил или запросил изменения
- [ ] Изменения внесены и перепроверены
